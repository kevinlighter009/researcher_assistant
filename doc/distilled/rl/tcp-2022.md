# Trajectory-guided Control Prediction for End-to-end Autonomous Driving: A Simple yet Strong Baseline

## Keywords
- il-rl-fusion, roach-distillation, trajectory-control-fusion, multi-step-control, gru-temporal, carla-leaderboard, monocular

## TL;DR
End-to-end driving methods split into trajectory-output (planning then PID) and direct control-output paradigms; each fails in characteristic ways (trajectory branch struggles in big turns due to PID following error; control branch reacts late to obstacles due to single-step IID assumption). The authors propose TCP, a multi-task framework with a shared image+measurement encoder feeding two branches: a GRU trajectory planner and a novel multi-step control branch distilled from the Roach RL expert, with the trajectory branch providing per-step attention guidance to the control branch. A situation-based fusion combines the two outputs. With only a monocular camera, TCP-Ens reaches Driving Score 75.137 on the public CARLA Leaderboard, ranking 1st and beating prior multi-camera+LiDAR methods by 13.291 points.

## Problem & Motivation
Prior end-to-end driving work splits cleanly into two camps. (1) Trajectory-based methods (Transfuser, NEAT, LAV, LBC) plan waypoints then run a PID controller; the trajectory-to-control conversion is non-trivial and PID performs poorly on inertial maneuvers like big turns or red-light starts. (2) Control-based methods (CIL, CILRS, Roach) directly regress throttle/brake/steer; they implicitly assume IID state-action pairs, focus on the current step only, suffer reaction latency to abrupt obstacles, and produce unstable/discontinuous actions. No prior work combines the two paradigms in a unified network — naive ensembling doubles model size and ignores cross-task correlations. Additionally, control branches are typically trained as pure behavior cloning, which violates the sequential-decision nature of driving.

## Innovation Points
- **Two-branch unified framework** — first work to jointly learn and fuse a trajectory branch and a direct control branch in a single network with shared encoder, framed as multi-task learning rather than ensemble.
- **Multi-step control branch** — instead of single-step BC, the control branch predicts K future actions auto-regressively via a GRU "temporal module" that simulates ego-environment interaction in latent space, breaking the IID assumption without resorting to RL/MDP.
- **Trajectory-guided attention** — at each future step t, hidden states from the waypoint GRU and the control GRU produce an attention map over the 2D image feature, letting the control branch attend to scene regions relevant to its future location (curbs, lanes, traffic lights).
- **Roach-distilled expert with feature/value/KL losses** — control branch is supervised against the Roach BEV-RL expert via Beta-distribution KL loss plus latent-feature loss and an auxiliary value head, importing RL-learned knowledge into a vision-only student.
- **Situation-based fusion** — Algorithm 1 chooses fusion weight alpha (in [0, 0.5]) based on whether the ego is turning: trajectory-specialized branch dominates on straights, control-specialized on turns.

## Model Architecture
- Input: monocular RGB image i, scalar speed v, navigation info g (high-level command + 2D goal coordinates).
- Image encoder: ResNet (ResNet-34 in ablations) producing 2D feature map F.
- Measurement encoder: MLP over concat(g, v) producing j_m.
- Shared features fed to two branches:
  - **Trajectory branch**: avg-pooled F + j_m -> j^traj -> GRU auto-regresses K future waypoints wp_1..wp_K (Transfuser-style); waypoints converted to throttle/brake/steer via two PID controllers (longitudinal magnitude + lateral orientation).
  - **Multi-step control branch**: temporal-module GRU rolls forward K steps; at each t, hidden states (h^traj_t, h^ctl_t) -> MLP -> attention map w_t in R^{1xHxW}; w_t softmax-weights F; concat with h^ctl_t -> j^ctl_t -> shared policy head outputs Beta-distribution action a_t (throttle in [0,1], brake in [0,1], steer in [-1,1]).
- Losses: L_traj (L1 waypoint + feature L2 to expert), L_ctl (KL between predicted and Roach Beta + per-step feature loss), L_aux (speed L1 + value L2). Total weighted sum.
- Fusion: situation-based weighted average a = alpha * a^ctl + (1-alpha) * a^traj, alpha in [0, 0.5]; alpha = 0.3 used.
- Expert: Roach (BEV-RL trained with privileged info on roads/lanes/routes/vehicles/peds/lights).
- Training data: 420K samples in 8 CARLA towns under random weather; ablations use 189K (Towns 01/03/04/06).
- Model size (Table 4): TCP 25.77M params, 8.54G FLOPs, 125.71 FPS. TCP-Ens 73.03M, 25.61G, 44.70 FPS.

Multi-step temporal module is the largest single jump (+7.9 DS); trajectory-guided attention adds +3.2; fusion adds +11.

**Control-only vs Trajectory-only failure mix (Table 2):** Trajectory-only collides with vehicles less (0.85 vs 1.25 /km) but suffers more layout collisions, off-road, and agent-block events that concentrate during turns (TurnRatio ~62-64% vs ~50%), confirming the PID-following weakness on turns motivating the fusion.

**MTL vs Ensemble (Table 4):** TCP (MTL) reaches DS 57.01 with 25.77M params at 125.71 FPS; naive Ensemble reaches 45.03 DS with 46.81M params at 69.47 FPS — MTL is both better and cheaper. TCP-Ens further boosts to 59.09 at the cost of compute.

## Limitations & Open Questions
- Evaluated only in CARLA (sim-only); no real-vehicle or real-world benchmark.
- Relies on a strong privileged-info RL expert (Roach); transfer of the recipe to settings without such an expert is not studied.
- Situation criterion is hand-engineered (binary "is turning"); the alpha is fixed at 0.3 — a learned/continuous gating would be a natural extension.
- Monocular-only input, but 3D-aware perception (BEV/depth) is not exploited; performance ceiling vs sensor-rich variants under harder weather/long-range obstacles is unclear.
- Single time-step image input (no temporal frame stacking on the perception side); temporal reasoning lives entirely in the latent GRU rollout.
