# VAD: Vectorized Scene Representation for Efficient Autonomous Driving

## Keywords
- vectorized-scene, end-to-end-planning, bev, planning-constraints, nuscenes, carla

## TL;DR
Prior end-to-end driving methods (ST-P3, UniAD) rely on dense rasterized scene representations (semantic/occupancy/cost maps), which are computationally heavy and lose instance-level structure. VAD encodes the scene as vectorized agent motion vectors and vectorized map elements (lane/boundary/crossing), and applies three explicit instance-level planning constraints (ego-agent collision, ego-boundary overstepping, ego-lane direction). On nuScenes, VAD-Base cuts average L2 from 1.03m (UniAD) to 0.72m and average collision rate from 0.31% to 0.22% while running 2.5x faster; VAD-Tiny runs 9.3x faster with comparable accuracy.

## Problem & Motivation
Modular AD pipelines decouple perception and planning, propagating perception error and discarding sensor-rich semantics. End-to-end methods that go directly from sensors to control (CILRS-style) lack interpretability. The dominant interpretable end-to-end family (ST-P3, UniAD) routes through dense rasterized intermediates (semantic map, occupancy map, flow map, cost map), which are (a) computationally expensive and (b) blur out the per-instance structure (lane direction, agent identity, boundary geometry) that planning actually needs. VAD argues a vectorized representation is both faster and a better planning prior.

## Innovation Points
- **Fully vectorized scene representation** — driving scene encoded as vectorized map vectors (lane divider, road boundary, pedestrian crossing) and per-agent multi-modality motion vectors, replacing rasterized semantic/occupancy/cost maps.
- **Planning via query interaction** — randomly initialized ego query interacts first with agent queries (ego-agent attention) then with map queries (ego-map attention) to implicitly absorb instance-level scene context before the planning head.
- **Three vectorized planning constraints** — explicit instance-level losses: ego-agent collision constraint (lateral/longitudinal safety thresholds vs closest agent), ego-boundary overstepping constraint (distance to closest map boundary), ego-lane directional constraint (angular alignment with closest lane vector).
- **HD-map-free, post-processing-free design** — no offline HD map; no hand-designed cost maps or rule-based trajectory selection. Driving command (turn left / turn right / go straight) is the only navigation input.
- **Two scale variants** — VAD-Base (full) and VAD-Tiny (smaller BEV grid 100x100, 3 transformer layers, 640x360 input) for the speed/accuracy trade-off.

## Model Architecture
- Inputs: multi-view multi-frame surround camera images; 2-second history; driving command (turn left / right / go straight); optional ego status.
- **Backbone**: ResNet-50 image encoder + BEV encoder (BEVFormer-style) projects image features into BEV features; perception range 60m x 30m (long. x lat.).
- **Vectorized Scene Learning**:
  - Map queries `Q_m` -> Vectorized Map Transformer -> map vectors `V_m in R^{N_m x N_p x 2}` (default `N_m=100`, `N_p=20`); classes: lane divider, road boundary, pedestrian crossing.
  - Agent queries `Q_a` -> Vectorized Motion Transformer with agent-agent and agent-map attention -> per-agent multi-modality motion vectors `V_a in R^{N_a x N_k x T_f x 2}` plus modality probabilities.
- **Planning Inferring**: ego query `Q_ego` -> Transformer decoder attending to agent queries (ego-agent interaction) -> attends to map queries (ego-map interaction) -> updated ego features concatenated with optional ego status and driving command -> MLP Planning Head -> ego trajectory `V_ego in R^{T_f x 2}` (3-second horizon).
- **Planning Training Phase**: vectorized planning constraints `L_col`, `L_bd`, `L_dir` regularize the predicted trajectory against the vectorized scene; total loss combines map, motion, three constraint, and L1 imitation losses.
- VAD-Tiny: BEV grid 100x100; map/agent/ego queries 100/20/300 (Base: 200x200, 6 layers); image input 640x360 (Base: 1280x720). VAD-Base also adds a ResNet-50 traffic-light classification branch on the cropped front view for closed-loop CARLA.

## Benchmark Results
**Open-loop planning, nuScenes val (no ego status):**

| Method     | L2 Avg (m) | Coll Avg (%) | Latency (ms) | FPS  |
|------------|------------|--------------|--------------|------|
| ST-P3      | 2.11       | 0.71         | 628.3        | 1.6  |
| UniAD      | 1.03       | 0.31         | 555.6        | 1.8  |
| VAD-Tiny   | 0.78       | 0.38         | 59.5         | 16.8 |
| VAD-Base   | **0.72**   | **0.22**     | 224.3        | 4.5  |

VAD-Base reduces avg L2 by 30.1% (1.03 -> 0.72) and avg collision rate by 29.0% (0.31 -> 0.22) vs UniAD, with 2.5x speedup. VAD-Tiny is 9.3x faster than UniAD with comparable accuracy. With ego-status input: VAD-Base L2 0.37m / Coll 0.14% (paper notes ego-status risks shortcut learning, kept for reference).

**Closed-loop CARLA Town05:**

| Method      | Town05 Short DS / RC | Town05 Long DS / RC |
|-------------|----------------------|---------------------|
| ST-P3       | 55.14 / **86.74**    | 11.45 / **83.15**   |
| VAD-Base    | **64.29** / 87.26    | **30.31** / 75.20   |

VAD-Base improves Town05 Short DS by 9.15 over ST-P3 and Town05 Long DS from 11.45 to 30.31 (close to LiDAR-based Transfuser's 33.15) for vision-only methods.

**Ablations (nuScenes, Tab. 2):** removing all three vectorized constraints (ID 3) increases avg collision from 0.22% to 0.28%; adding constraints individually (IDs 4-6) all reduce collision; ego-map interaction is critical (ID 1 without map interaction -> avg L2 0.86m vs 0.72m). **Map representation (Tab. 3):** rasterized map variant gives L2 0.74 / Coll 0.39 vs vectorized 0.72 / 0.22 — vectorized representation roughly halves the collision rate.

**Module runtime (VAD-Tiny, RTX 3090):** Backbone 23.2ms (39.0%), BEV Encoder 12.3ms (20.7%), Motion 11.5ms (19.3%), Map 9.1ms (15.3%), Planning 3.4ms (5.7%); total 59.5ms.

## Limitations & Open Questions
- Multi-modality motion is predicted per agent but the collision constraint uses only the highest-confidence trajectory — exploiting full modality distribution for safer planning is left as future work.
- Vectorized scene currently covers lane divider, road boundary, and pedestrian crossing only; lane graph topology, road signs, traffic lights (beyond a separate classifier branch), and speed limits are not integrated.
- Open-loop nuScenes L2/collision metrics have known shortcut-learning issues with ego status; closed-loop validation is limited to CARLA Town05.
- Planning constraint thresholds (`delta_X=1.5m`, `delta_Y=3.0m`, `delta_bd=1.0m`, `delta_dir=2.0m`) are hand-tuned hyperparameters whose generalization across geographies/vehicle classes is not analyzed.
